# Графік випуску креслень

Вебверсія Excel-графіка для публічного перегляду та захищеного редагування. Інтерфейс містить 68 початкових креслень, пошук і фільтри, календар Ганта, кольорові позначки виконавців та автоматичний розрахунок завершення в робочих днях.

## Можливості

- публічний перегляд без входу;
- редагування після введення пароля адміністратора;
- зміна всіх полів, додавання, видалення та переставляння рядків;
- ручне планування або автоматичний початок після завершення кількох попередніх робіт;
- стабільні залежності за номерами робіт, які не розриваються після перенумерації;
- каскадний перерахунок багаторівневих залежностей із захистом від циклів і випадкового видалення попередників;
- підсвічування всього ланцюжка попередників і наступників;
- лінія сьогоднішньої дати, приглушений минулий календар і сірі минулі частини робіт;
- порівняння поточного графіка з будь-якою з 10 попередніх збережених версій;
- керування виконавцями, їхніми назвами та кольорами;
- збереження даних у Cloudflare D1;
- Excel-подібні кольори: блакитне чергування рядків, рожеві вихідні та окремий колір кожного виконавця;
- автоматичний прогрес кожного листа, розділу та всього графіка, зважений за робочими днями;
- адаптивний вигляд із горизонтальним прокручуванням таблиці на телефоні.

## Розрахунок прогресу

Лист із датою початку та тривалістю починається з `0%` у день старту. Кожен наступний робочий день додає частку відповідно до тривалості; субота й неділя не враховуються. Незавершений лист обмежується `95%`, а статус «Завершено» одразу дає `100%`. Листи без дати або тривалості не входять до підсумків.

## Додаткові вихідні

Глобальні неробочі дати задаються вручну у `public/holidays.json` як JSON-масив дат у форматі `YYYY-MM-DD`:

```json
["2026-08-24", "2026-12-25"]
```

Дублікати та некоректні значення ігноруються. Після зміни файла потрібно повторно розгорнути статичні ресурси й перезавантажити сторінку. Додаткові вихідні не враховуються в тривалості, залежностях і прогресі та позначаються на шкалі Ганта тим самим кольором, що й суботи та неділі.

Прогрес розділів і всього графіка зважується за тривалістю листів. Список розділів формується автоматично з даних графіка, тому новий розділ з'являється в панелі без додаткового налаштування. Фільтри змінюють лише видимі рядки й не спотворюють загальний або секційні підсумки.

## Локальний запуск

Потрібен Node.js 20 або новіший.

```powershell
npm install
Copy-Item .dev.vars.example .dev.vars
npm run build
npx wrangler d1 migrations apply logistruct-schedule-db --local
npm run dev:cf
```

Після запуску відкрийте адресу, яку покаже Wrangler. Пароль задається у `.dev.vars` змінною `ADMIN_PASSWORD`.

Швидкі перевірки:

```powershell
npm test
npm run lint
npm run build
```

## Тестовий сервер

Окремий тестовий сервер використовує всі 68 креслень зі списку та наповнює їх демонстраційними датами, статусами, виконавцями, залежностями й історією версій. Його D1-стан зберігається в `.wrangler/test-state` і не змінює звичайну локальну базу в `.wrangler/state`.

Створити чисту тестову базу та запустити сервер:

```powershell
npm run db:test:reset
npm run dev:test
```

Сервер буде доступний на `http://127.0.0.1:8788`. Команда `db:test:reset` видаляє лише `.wrangler/test-state`, повторно застосовує міграції й створює демонстраційний набір відносно поточної дати. Щоб відновити набір без видалення тестової бази, виконайте:

```powershell
npm run db:test:seed
```

Обидві команди працюють тільки з локальним D1 і ніколи не використовують `--remote`.

## Розгортання на Cloudflare Pages

Проєкт розрахований на Pages Functions і D1 та може працювати в межах безкоштовних лімітів Cloudflare для невеликого робочого графіка.

1. Увійдіть у Cloudflare:

   ```powershell
   npx wrangler login
   ```

2. Створіть базу та скопіюйте виданий `database_id`:

   ```powershell
   npx wrangler d1 create logistruct-schedule-db
   ```

3. Додайте `"database_id": "отриманий-id"` до об'єкта `d1_databases` у `wrangler.jsonc`.

4. Створіть Pages-проєкт:

   ```powershell
   npx wrangler pages project create logistruct-schedule
   ```

5. У Cloudflare Dashboard відкрийте **Workers & Pages → logistruct-schedule → Settings → Variables and Secrets** та додайте зашифровані секрети:

   - `ADMIN_PASSWORD` — ваш складний пароль адміністратора;
   - `SESSION_SECRET` — довгий випадковий рядок, щонайменше 32 символи.

6. Застосуйте початкову міграцію до віддаленої D1:

   ```powershell
   npx wrangler d1 migrations apply logistruct-schedule-db --remote
   ```

7. Зберіть і опублікуйте сайт:

   ```powershell
   npm run deploy
   ```

Після публікації сайт буде доступний за адресою `https://logistruct-schedule.pages.dev`. Якщо назва вже зайнята, використайте іншу назву проєкту і таку саму назву в команді `pages deploy` у `package.json`.

### Оновлення існуючого сайту

Перед публікацією нової версії завжди застосовуйте всі невиконані міграції. Міграція `0003_dependencies_history.sql` додає режими початку, нормалізовані залежності та історію десяти попередніх станів:

```powershell
npx wrangler d1 migrations apply logistruct-schedule-db --remote
npm run deploy
```

## Залежності та історія

У режимі редагування поле «Початок за» дозволяє обрати «Датою» або «Залежностями». Для автоматичного режиму виберіть один чи кілька номерів робіт; дата початку дорівнюватиме найпізнішій даті їх завершення. У базі зв’язок зберігається за незмінним ідентифікатором, тому переставляння рядків лише оновлює показані номери. Цикли блокуються, а роботу-попередник не можна видалити, доки залежності від неї не очищено вручну.

Кнопка аналізу в рядку підсвічує весь ланцюжок у двох напрямках. Кнопка «Порівняти» завантажує список до десяти попередніх збережень і показує старі смуги контурами, змінені комірки маркерами, додані роботи зеленим, а видалені — в окремій червоній секції. Порівняння виконується лише для збереженого стану; незбережені зміни спочатку потрібно зберегти або скасувати.

## Структура

- `src/` — React-інтерфейс і логіка графіка;
- `functions/api/` — Pages Functions для входу та збереження;
- `migrations/` — схема D1 і початкові 68 рядків;
- `tests/` — модульні й API-тести;
- `docs/` — погоджений дизайн та план реалізації.

Локальні `.dev.vars`, дані Wrangler і згенеровані файли не додаються до Git.
