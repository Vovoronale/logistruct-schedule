# Logistruct Schedule Site Design

**Date:** 2026-07-03

**Goal:** Build a polished public construction-drawing schedule that resembles the useful part of the Excel workbook, supports password-protected editing, and can run on the Cloudflare free tier using Pages, Pages Functions, and D1.

## Scope

The site presents the 68 drawing records from `ГРАФІК LOGISTRUCT_оновлений.xlsx` as its initial dataset. Anyone can view the schedule. A single administrator can sign in with one password and edit the schedule, add or delete rows, and reorder them.

The website intentionally omits Excel's technical cells and helper columns. Dependency formulas, internal counters, export ranges, and formula-debugging fields are not exposed in the interface.

## Architecture

- A React and TypeScript single-page application is built with Vite and served as static assets by Cloudflare Pages.
- Cloudflare Pages Functions provide the public read API, authentication endpoints, and authenticated write API.
- Cloudflare D1 stores schedule rows and the current dataset revision.
- The initial D1 migration seeds the exact 68 records from the workbook. Dates, durations, assignees, and statuses start empty.
- The public page requires no account or cookie.
- Administrator sessions use a signed, `HttpOnly`, `Secure`, `SameSite=Strict` cookie with a 12-hour lifetime.
- `ADMIN_PASSWORD` and `SESSION_SECRET` are configured as Cloudflare secrets and never committed to the repository or returned to the browser.

## Data Model

Each schedule item contains:

- `id`: stable UUID generated by the server;
- `position`: integer display order;
- `section`: drawing section, such as `КЗ-0` or `КМ2`;
- `sheetNumber`: integer sheet number within the section;
- `title`: drawing name;
- `startDate`: optional ISO date;
- `durationDays`: optional positive integer measured in working days;
- `assignee`: optional short assignee code;
- `status`: `planned`, `in_progress`, or `completed`;
- `createdAt` and `updatedAt`: server timestamps.

The database also stores a single monotonically increasing schedule revision. Every public response includes the revision. A save must include the revision that the administrator originally loaded; a mismatch returns a conflict instead of overwriting newer data.

The end date is derived from `startDate` and `durationDays`. Saturdays and Sundays do not count as working days, matching the workbook's `WORKDAY.INTL(..., 1)` behavior. Rows with an incomplete start date or duration show no calculated end date or Gantt bar.

## Public Interface

The main page uses a clean, bright spreadsheet-inspired layout with a sticky header and frozen identifying columns. The visible columns are:

`№ · Розділ · № листа · Найменування креслення · Початок · Робочі дні · Завершення · Виконавець · Статус`

A horizontally scrollable Gantt calendar appears to the right of the table. It includes month and day headers, highlights weekends, and draws each scheduled item as a colored bar. The visible date range is calculated from scheduled rows with modest padding before the earliest start and after the latest finish. If no dates exist, the timeline displays a concise empty state while the table remains fully usable.

The public toolbar contains:

- title and last-updated timestamp;
- drawing-name search;
- section, assignee, and status filters;
- a compact assignee color legend;
- the administrator login/edit button.

Filtering affects both the table and aligned Gantt rows. The table and timeline maintain synchronized vertical scrolling. On narrow screens, the schedule uses horizontal scrolling rather than collapsing data into ambiguous cards.

## Visual Language and Excel Color Mapping

The visual system preserves the workbook's recognizable conditional-formatting palette while presenting it with modern spacing, typography, borders, controls, and subtle shadows.

## Administrator Experience

Selecting “Редагувати” opens a small password dialog. A successful login switches the same page into edit mode without exposing a separate administration URL.

In edit mode:

- editable table cells use appropriate controls for text, dates, positive integers, assignees, and statuses;
- rows can be added, deleted after confirmation, and reordered with drag handles plus keyboard-accessible move controls;
- calculated end dates and Gantt bars update immediately in the local draft;
- “Зберегти зміни” submits the entire ordered draft with its original revision;
- “Скасувати” restores the last saved dataset after confirmation when changes exist;
- leaving the page with unsaved changes triggers a browser warning.

The server validates every submitted row, normalizes ordering, and writes the new dataset and incremented revision atomically. The public dataset changes only after the entire save succeeds.

## API and Authentication

- `GET /api/schedule` returns the public schedule, revision, and last-updated timestamp.
- `POST /api/auth/login` verifies the administrator password and creates the signed session cookie.
- `POST /api/auth/logout` clears the session cookie.
- `GET /api/auth/session` reports whether the current request has a valid administrator session.
- `PUT /api/schedule` validates an authenticated full-dataset replacement and expected revision, then commits it atomically.

Authentication failures return generic messages that do not reveal password details. Mutating endpoints reject missing, invalid, or expired sessions. Request bodies have conservative size limits, and all database values are passed through parameterized D1 statements.

## Error Handling

- Initial load failures show a retry action without rendering misleading empty data.
- Failed saves retain the administrator's complete local draft and display an actionable error message.
- A revision conflict explains that newer data exists and offers reload rather than silently overwriting it.
- Validation errors identify the affected row and field.
- The public view always reflects the last successfully committed dataset.
- Network and server errors are logged without including passwords or session secrets.

## Testing and Acceptance Criteria

Automated checks cover:

- working-day end-date calculation across weekends and invalid inputs;
- request validation and ordering normalization;
- session signing, expiry, and protected-route rejection;
- public schedule reads and atomic authenticated saves;
- revision-conflict behavior;
- filtering, editing, adding, deleting, and reordering rows;
- Gantt alignment and date-range generation;
- administrator login, save, cancel, logout, and unsaved-change warning;
- responsive table behavior and key accessibility flows.

The implementation is accepted when:

1. A fresh D1 database contains all 68 supplied drawing records in the correct order.
2. Visitors can view and filter the schedule without authenticating.
3. Only a valid administrator session can mutate data.
4. A successful edit is visible to a subsequent public request.
5. End dates exclude Saturdays and Sundays.
6. The Gantt bars and row markers use the extracted Excel palette.
7. The production build and automated test suites pass.
8. The repository documents local development, D1 migration, secret configuration, and Cloudflare Pages deployment commands without requiring a paid Cloudflare plan.

## Deployment and Operations

The repository includes Wrangler configuration, D1 migrations, a sanitized environment example, and README instructions for local development and Cloudflare Pages deployment. No credentials, database identifiers from another account, or local environment files are committed. The deployment flow supports the Workers Free plan and does not require a paid Cloudflare subscription for the intended small-team workload.
