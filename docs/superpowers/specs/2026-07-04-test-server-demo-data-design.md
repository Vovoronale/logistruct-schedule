# Ізольований тестовий сервер із демонстраційними даними

## Мета

Створити локальний тестовий сервер, який використовує всі 68 креслень із наявного Excel-списку, але не змінює поточну локальну D1-базу в `.wrangler/state`. Набір має давати змогу одразу перевіряти фільтри, прогрес, виконавців, ручне й залежне планування, ланцюжки залежностей та порівняння версій.

## Ізоляція

Тестове середовище зберігатиметься в `.wrangler/test-state` через підтримуваний Wrangler-параметр `--persist-to`. Міграції, наповнення та `wrangler pages dev` завжди отримуватимуть однаковий шлях. Поточна база `.wrangler/state`, віддалена D1 і production-конфігурація не змінюються.

`.wrangler/test-state` є локальним згенерованим станом і не потрапляє до Git. У репозиторії зберігаються лише відтворюваний генератор тестових даних, його тести та команди запуску.

## Команди

- `npm run db:test:reset` — видаляє лише `.wrangler/test-state`, застосовує всі міграції та наповнює нову тестову базу.
- `npm run db:test:seed` — повторно приводить вже створену тестову базу до визначеного демонстраційного стану без впливу на звичайну локальну базу.
- `npm run dev:test` — збирає застосунок і запускає Pages Functions із тестовим D1-станом на окремому порту `8788`.

Команда скидання має перевіряти абсолютний шлях перед видаленням і відмовлятися працювати, якщо ціль не дорівнює очікуваній директорії `.wrangler/test-state` всередині репозиторію.

## Профіль даних

Базові назви, розділи, номери листів та порядок усіх 68 креслень беруться зі schema seed у `migrations/0001_initial.sql`, який уже походить із Excel-файлу. Генератор змінює лише тестові атрибути:

- усі 17 виконавців використовуються щонайменше один раз;
- присутні `planned`, `in_progress` і `completed`;
- частина запланованих робіт лишається без дат і виконавців для перевірки порожніх станів;
- дати формуються відносно дня запуску, щоб у графіку одночасно були минулі, поточні та майбутні роботи;
- тривалості охоплюють короткі, середні та довгі роботи, включно з переходами через вихідні;
- створюються кілька незалежних ланцюжків, розгалуження та об'єднання залежностей без циклів;
- залежні роботи використовують `start_mode = 'dependencies'`, а решта — ручний режим;
- створюються три попередні ревізії в `schedule_history`, щоб порівняння показувало змінені дати, статуси, виконавців, доданий і видалений елементи;
- поточний `schedule_meta.revision` узгоджується з історією.

Генерація детермінована для однакової опорної дати. Опорна дата за замовчуванням — поточний локальний день; у тестах її можна зафіксувати параметром.

## Компоненти та потік

Окремий модуль генератора формує типізовану модель fixture: оновлення креслень, залежності, історичні snapshots і метадані. CLI-обгортка перетворює модель на D1 batch-файли до 80 КБ і виконує їх тільки проти ізольованого D1 state. Спочатку перевіряються кількість і незмінні ідентифікатори 68 базових записів; при невідповідності наповнення припиняється з поясненням.

Повторний запуск є ідемпотентним: тестові залежності та історія замінюються, а атрибути існуючих 68 креслень повертаються до визначеного fixture. Нові дублікати не накопичуються.

## Обробка помилок і безпека

- жодна команда не використовує `--remote`;
- шлях persistence передається явно в кожному D1/Pages виклику;
- кожен SQL batch виконується у власній неявній транзакції D1; ручні `BEGIN` і `COMMIT`, які D1 не підтримує, не генеруються;
- генератор екранує текстові значення та перевіряє посилання залежностей;
- цикл, відсутній ідентифікатор або неправильна кількість базових записів зупиняють запуск до зміни бази;
- секрети беруться з наявного локального `.dev.vars` і не виводяться та не комітяться.

## Перевірка

Автоматичні тести перевіряють детермінованість, розподіл усіх статусів і виконавців, коректність дат, відсутність циклів, узгодженість ревізій та SQL-екранування. Після наповнення окремі D1-запити підтверджують 68 поточних записів, три статуси, 17 виконавців, наявність залежностей і трьох історичних ревізій.

Фінальний smoke-test запускає `dev:test`, відкриває `http://127.0.0.1:8788`, перевіряє завантаження 68 рядків, відсутність помилок консолі та щонайменше такі взаємодії: пошук, фільтр статусу, відкриття аналізу залежностей і вибір попередньої ревізії для порівняння.
